# Playwright × Playwright MCP ハイブリッドテスト戦略

> **関連 Issue**: [#45](https://github.com/junichi-muraoka/jcsqe-study-app/issues/45)（任意・低優先度）

## 導入状況

| 項目 | 状態 |
|------|------|
| **Playwright（CI）** | 導入済み（[`.github/workflows/e2e.yml`](../.github/workflows/e2e.yml)、[`tests/e2e.spec.js`](../tests/e2e.spec.js)） |
| **Playwright MCP** | **未導入**（任意）。AI による探索・テスト生成を行う場合のみ Cursor 等に MCP サーバーを追加する。優先度は低く、CI の E2E だけで回帰を担保できる間は後回しでよい。 |

Playwright（CI用E2E）と Playwright MCP（AI駆動）の両方のメリットを活かす構成案です。

---

## 設計思想

| フェーズ | 担当 | 目的 |
|---------|------|------|
| **テスト作成** | Playwright MCP（AI） | 柔軟な探索・自然言語での操作・テストコード生成 |
| **テスト実行** | Playwright（CI） | 決定論的・再現可能・PRごとの自動検証 |

**「AIがテストを書く → CIがテストを実行する」** の分業で、両方の強みを組み合わせる。

---

## 推奨ワークフロー

```
┌─────────────────────────────────────────────────────────────────┐
│ 開発中（ローカル / Cursor）                                        │
├─────────────────────────────────────────────────────────────────┤
│  1. 新機能を実装                                                  │
│  2. AIに依頼: 「Playwright MCPでこの機能をテストして、            │
│     tests/e2e.spec.js に追加するテストコードを生成して」          │
│  3. AIがPlaywright MCPでブラウザ操作 → 成功した操作を             │
│     Playwrightテスト形式で出力                                    │
│  4. 開発者が生成コードをレビュー・調整して tests/ に追加            │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│ PR作成・マージ時（GitHub）                                        │
├─────────────────────────────────────────────────────────────────┤
│  5. 既存の .github/workflows/e2e.yml が自動実行                   │
│  6. 決定論的なPlaywrightテストがCIでグリーン/レッド判定             │
│  7. リグレッションを検知                                           │
└─────────────────────────────────────────────────────────────────┘
```

## 具体的な使い方

### シナリオA: 新機能のテストを追加したい

**開発者**: 「弱点克服モードのフローをテストするPlaywrightコードを生成して。既存の tests/e2e.spec.js の形式に合わせて」

**AI（Playwright MCP利用時）**:
1. アプリを開く
2. 「弱点克服」をクリック
3. 問題が出るか確認
4. 1問解答
5. 上記を `test('弱点克服モードで...')` 形式のコードで出力

**開発者**: 出力を `tests/e2e.spec.js` に追記 → PR

### シナリオB: リグレッションを検知したい

**開発者**: 何もしない（既存CIが対応）

**CI**: PRごとに `npm run test:e2e` を実行。失敗したらマージブロック。

### シナリオC: 探索的に動作確認したい

**開発者**: 「ブックマーク機能が正しく動くか、Playwright MCPで確認して」

**AI**: ブラウザを操作してブックマークのON/OFF、一覧表示などを確認。問題があれば報告。

## メリット・デメリット（ハイブリッド構成）

### メリット
- **テスト作成の負荷軽減**: AIが探索してコードを生成
- **CIの安定性**: 実行は従来のPlaywrightで決定論的
- **網羅性の向上**: 人間が思い付かないシナリオをAIが提案
- **段階的導入**: Phase 1 だけで十分な場合、MCPは省略可能

### デメリット
- **MCP設定が必要**: Phase 2 以降は環境構築が必要
- **生成コードのレビュー**: AI出力は必ず人間が確認する必要がある
- **二重メンテ**: Playwright と Playwright MCP の両方を理解する必要がある場合も

## まとめ

| 用途 | 使うツール |
|------|-----------|
| PR時のリグレッション検知 | **Playwright（CI）** ← 現状のまま |
| 新機能のテストコード生成 | **Playwright MCP** ← 必要に応じて導入 |
| 探索的な動作確認 | **Playwright MCP** |
| 安定した自動テスト実行 | **Playwright（CI）** |

**最小構成**: 現状の Playwright CI のみで運用可能。  
**拡張構成**: Playwright MCP を追加し、AI にテスト生成を任せる。
