## 概要

アプリの利用を **Firebase（Google）ログイン必須** とし、初回・未ログイン時は **専用のログイン画面** を表示する。現状の「オフライン優先・ログインは任意（設定から）」から方針を切り替える。

## 背景・現状

- 学習データは主に `localStorage`。クラウド同期は **Firebase + Firestore**（`firebase-config.js` 設定時）。
- ログインは **設定タブ**から任意。未ログインでも問題演習・成績表示は可能。
- Service Worker によりオフライン利用も想定。

## 目的

- **利用者とクラウド上のデータを常に対応**させ、サポート・バックアップ説明を単純化する。
- **未ログインのまま進捗だけ端末に残る**状態をなくし、意図しないデータ分断を減らす。

## ユーザー体験（案）

| シナリオ | 挙動（案） |
|----------|------------|
| 初回アクセス | **ログイン画面のみ**（またはホームを隠しログインを強制）。Google でログイン成功後にホームへ。 |
| ログイン済み再訪問 | そのままホーム（トークン更新は Firebase SDK に任せる）。 |
| ログアウト | 再びログイン画面へ。**localStorage の扱いは要決定**（下記）。 |
| `firebase-config.js` 未設定 | ログイン機能が使えないため、**開発者向けメッセージ**またはビルド時エラー。本番では設定必須をドキュメント化。 |

## 技術的な検討ポイント

### 1. 画面ガード

- **Firebase Auth の `onAuthStateChanged`** で `user` が null の間は、メイン UI（ナビ・各タブ）をマウントしない／非表示にする。
- **専用スクリーン**（例: `#screen-login`）にアプリ名・簡単な説明・「Google でログイン」ボタンを配置。既存の `firebase-signin-btn` ロジックを移設または共有。

### 2. 既存 localStorage データとの関係（要決定）

- **案 A**: ログイン後、既存の `localStorage` データを **そのまま** 使い、以降の `saveData` で Firestore と同期（初回マージルールは `firebase-sync.js` 既存の考え方に合わせる）。
- **案 B**: ログイン時に「端末に古いデータがあります」と警告し、**マージ / 上書き / 破棄**を選択させる。
- **案 C**: ログイン必須導入後は **新規 UID 前提**で空から始める（厳しめ。既存ユーザーへの影響大）。

→ **案 A または B** を推奨（既存利用者への衝撃を抑える）。

### 3. オフライン

- ログイン必須でも、**一度ログインした後**は Firebase がトークンを保持する範囲でオフライン利用は可能な場合があるが、**初回・再認証**にはネットワークが必要。
- **完全オフラインで初回から使う**ニーズは捨てる前提になる。README・免責で明記する。

### 4. セキュリティ・プライバシー

- 共有端末では **ログアウトの明示**がより重要になる（設定にログアウトを残す、ログイン画面からも案内）。
- [docs/security.md](docs/security.md) を更新。

## リスク・デメリット（合意形成用）

- **離脱率**: 開いてすぐログイン要求は学習アプリでは障壁になりやすい。
- **Google アカウントを持たない／使いたくない利用者**は利用不可になる。
- **会社・制限ネットワーク**で Google ログインがブロックされる環境では使えない。

## 受け入れ条件（案）

- [ ] 未ログイン時、**問題演習・成績・用語集などメイン機能に入れない**（ログイン画面以外を見せない、または操作不可）。
- [ ] ログイン成功後、従来どおり学習・Firestore 同期が動作する（設定は `firebase-config.js` 前提）。
- [ ] ログアウト後は再びログイン画面に戻る。
- [ ] `docs/firebase_manual_setup.md` / README に「**本アプリはログイン必須**」と前提を追記。
- [ ] （任意）E2E または手動テスト手順を Runbook に1行追記。

## 非スコープ（この Issue では決めない／後続でも可）

- メール／パスワードログインの追加。
- ログイン画面のデザインリニュー（まずは既存コンポーネント流用で可）。

**結論**: ログイン必須化は **実装可能**だが、**既存 localStorage の扱い**と **オフライン方針の説明**を決めたうえで着手するのが安全。UI は **専用ログイン画面 + Auth ガード** が最小構成。
